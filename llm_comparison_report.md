# 📄 LLM 모델 비교·선정 보고서

이 문서는 동일한 업무 자동화 과업에 대해 서로 다른 3종 이상의 LLM(대규모 언어 모델)의 성능을 객관적으로 비교하고, 최종적으로 업무에 가장 적합한 최적의 모델을 선정하는 보고서입니다.

## 1. 비교 목적
* **목표 과업**: 비정형 회의록 메모를 바탕으로 결정사항·할 일·확인 필요 섹션으로 구성된 마크다운 표 형식의 비즈니스 요약본을 자동 생성하는 과업 (회의록 요약 / 이메일 초안 작성)
* **비교 목적**: 지정한 업무 과업의 특성(마크다운 표 구조 출력, 가명 처리, 사족 차단, 한국어 톤 유지)에 따라, 가장 우수한 성능과 효율성을 제공하는 최적의 LLM을 선정하고자 함.

## 3. 테스트에 사용한 동일 입력 프롬프트 (Zero-shot / 원본)
> 세 모델에 동일하게 입력하여 응답 성능을 비교하기 위해 설계된 Zero-shot 프롬프트와 테스트용 회의록 데이터입니다. 아래 프롬프트를 그대로 복사하여 각 모델에 입력해 보세요.

```text
당신은 아래의 페르소나에 맞춰 동작하는 업무 요약 비서입니다. 아래 제공된 [회의 원문]을 바탕으로 다음 작성 규칙에 맞춰 요약본을 작성해줘.

[페르소나 설정]
* 이름: 아우토(Auto)
* 역할 및 전문 분야: 10년 경력의 프로젝트 관리(PMO) 전문가 및 회의록 분석가
* 대화 말투 및 톤: 군더더기 없는 두괄식 비즈니스 격식 경어체 사용. 답변 시작 전/후에 인사말, 잡담, 피드백 요구 등 불필요한 친근감 표출(사족)을 전면 금지하며 비즈니스 아웃풋 문서만 단독으로 반환.

[작성 규칙]
1. 결정사항(Decision)과 할 일(Action Items)을 명확히 구분하여 정리할 것.
2. 할 일은 담당자(실명 대신 직급/역할명 사용)와 구체적인 마감 기한을 명시할 것.
3. 정보가 누락되었거나 불확실하여 내부적으로 추가 확인이 필요한 사항은 '확인 필요' 섹션에 따로 기재할 것.
4. 말투는 격식 있는 비즈니스 한국어 경어체(~입니다, ~바랍니다)를 사용할 것.

[회의 원문]
"어제 진행된 스마트 스토어 고도화 프로젝트 회의 내용 공유합니다.
김철수 팀장님이 모바일 메인 페이지 시안은 다음 주 목요일까지 디자인 파트에 넘겨서 확정 짓자고 하셨습니다.
그리고 이영희 대리가 시스템 연동 API 명세서 작성하기로 했는데, 일정은 대충 다음 달 초까지는 어떻게든 마무리하겠다고 했습니다.
개발 서버 인프라 구축 예산은 지난번에 합의한 1,200만 원 범위 내에서 맞추는 거로 최종 결정 났습니다.
기타 상세 보안 규정에 대해서는 아직 인사팀 가이드라인이 안 내려와서 확인이 더 필요한 상황입니다."
```

## 4. 평가 축 기준 및 모델별 평가 결과

### 4.1. 평가 축 정의 (최소 4개 이상)
* **평가 축 1 - 정확성 및 논리성**: 팩트 왜곡이나 환각 없이 회의록 내용(결정사항·할 일·확인 필요 항목)을 올바르게 분류·반영했는가? 원문에 없는 정보를 창작하지 않았는가?
* **평가 축 2 - 한국어 자연스러움**: 번역체 느낌 없이 자연스러운 격식 비즈니스 한국어 경어체(~입니다, ~바랍니다)를 유지하는가?
* **평가 축 3 - 형식/템플릿 준수**: 마크다운 표(Table) 양식으로 결정사항·할 일·확인 필요 3개 섹션을 정확히 출력하는가? 사족(인사말·감사 멘트 등)이 완전 차단되는가?
* **평가 축 4 - 긴 문맥 유지력**: 10턴 멀티턴 대화 상황에서 이전 턴의 제약 조건(예산 제외, 가명 처리, 이메일 톤 등)을 누락 없이 지속적으로 기억·적용하는가?
* **평가 축 5 - 환각 차단 및 보안 필터링**: 기밀 수치·미정 정보·원문 미기재 사실에 대해 임의 창작 없이 답변을 차단하고, 적절한 후속 조치를 안내하는가?

## 5. 모델별 결과 요약 및 정성 평가

### 5.1. 모델 A (Claude Opus 4.6) 결과 분석
* **수행 결과 요약 (Zero-shot Raw 기준)**: Zero-shot 프롬프트 최초 입력 시 결정사항(`D-1/D-2`), 할 일(`A-1/A-2`), 확인 필요(`C-1/C-2`) 세 섹션을 마크다운 표 형식으로 완벽하게 구조화하여 출력함. 확인 필요 섹션에는 사유와 확인 대상 조직까지 세부 열로 구분하여 기재함. (Raw 참조: PART 1 [1])
* **강점**:
  - **시각적 가시성 극대화**: 마크다운 테이블을 완벽하게 사용하여 정보의 구분이 매우 명확함.
  - **정교한 가명 처리**: `김철수`, `이영희` 실명을 `팀장 (프로젝트 총괄)`, `대리 (시스템 연동 담당)`으로 역할 기반 비즈니스 호칭으로 자율 정제함.
  - **확인 필요의 세부 구조화**: 단순 나열을 넘어 사유·협의 대상을 별도 열로 구분하고 구체적인 대응 가이드를 기재함.
  - **10턴 멀티턴 누적 제약 완벽 유지**: 예산 수치 제외, 이메일 톤 전환, 마감 기한 취소선/굵은글씨 변경 이력 표기, 보안 규정 항목 제거 등 각 턴의 제약이 후속 턴에 완전히 누적 적용됨. (Raw 참조: PART 4 Turn 2~10)
* **약점 및 한계**:
  - Zero-shot 최초 입력 시 상대적 기한("다음 주 목요일", "다음 달 초")을 구체적 캘린더 날짜(YYYY-MM-DD)로 직접 역산하지 못함.
  - Turn 10 최종 이메일에서 승인 감사 인사("최종 승인 감사합니다!" 등) 일부 사족이 관측되었으나, 페르소나 설정 정신에서 일시적으로 벗어난 경미한 수준임.

### 5.2. 모델 B (Gemini 3.1 Pro) 결과 분석
* **수행 결과 요약 (Zero-shot Raw 기준)**: Zero-shot 입력에서 결정사항·할 일·확인 필요 항목을 적절히 분류하였으나, 지정한 마크다운 표 양식이 아닌 텍스트 볼드(`**`)와 불릿 기호(`*`)를 조합한 리스트 형태로 출력함. 가명 처리와 리스크 적발은 수행함. (Raw 참조: PART 1 [2])
* **강점**:
  - **가명 처리 및 리스크 파악**: 실명을 배제하고 `팀장`, `대리`로 가명화하였으며, 보안 규정 및 API 명세서 기한 모호성을 적절히 지적함.
* **약점 및 한계**:
  - **마크다운 표 포맷 준수 실패**: 명시적으로 표(Table) 형식을 요구했음에도 일반 텍스트 리스트 형태로 출력함. 자동화 시스템 연동 시 파싱 오류 발생 위험.
  - **노이즈 텍스트(사족) 포함**: 학습 팁, 서론 등 프롬프트 자동화 파이프라인 연동에 방해가 되는 불필요한 사족이 혼재됨.

### 5.3. 모델 C (GPT 5) 결과 분석
* **수행 결과 요약 (Zero-shot Raw 기준)**: 사족 없이 즉시 요약 텍스트를 출력하였으며, 회의 원문의 상대 기한을 실제 캘린더 날짜로 역산하는 능력이 독보적이었음. "다음 주 목요일"을 테스트일(2026-05-28) 기준 **`2026-06-04(목)`**로, "다음 달 초"를 **`2026-06-10(임시)`**으로 구체화하여 기재함. (Raw 참조: PART 1 [3])
* **강점**:
  - **초기 단발성 날짜 계산 우수(Zero-shot)**: 상대적 기한을 구체적 캘린더 날짜로 역산하는 성능이 3개 모델 중 유일하게 Zero-shot에서 구현됨.
  - **사족 없는 깔끔한 초기 출력**: 텍스트 시작 시 불필요한 인사말 없이 바로 요약 본문 출력.
* **약점 및 한계 (감점 요인)**:
  - **마크다운 표 포맷 준수 실패**: 마크다운 표 서식 없이 일반 텍스트 불릿 포인트 및 들여쓰기 리스트로 출력함.
  - **가명화 한계**: 담당자를 `팀장(전달)`, `대리`와 같이 원문 직급 그대로 1차원 매핑하는 데 그쳤으며, `프로젝트 팀장` 등 외부 공유에 적합한 역할 호칭으로 정제하지 못함.
  - **결정사항/할일 경계 혼재**: "모바일 메인 페이지 시안 전달"을 결정사항과 할 일 양쪽에 이중 기재하여 구분이 불명확함.

## 5.4. Claude Opus 4.6 단계별 프롬프트 비교 (Zero-shot vs Few-shot)
> Raw 실행 로그(PART 1 [1], PART 2 [1])를 기반으로, 최종 선정 모델인 Claude Opus 4.6의 Zero-shot(v1)과 Few-shot(v1.5) 단계 간 실제 아웃풋 변화를 항목별로 비교한 결과입니다.

### 📄 Zero-shot 실제 출력 (PART 1 [1] Raw)

```markdown
# 스마트 스토어 고도화 프로젝트 회의 요약

## 1. 결정사항 (Decision)

| No. | 내용 |
|-----|------|
| D-1 | 모바일 메인 페이지 시안을 디자인 파트에 전달하여 확정하기로 결정하였습니다. |
| D-2 | 개발 서버 인프라 구축 예산은 기합의 금액인 **1,200만 원** 범위 내에서 집행하는 것으로 최종 확정되었습니다. |

## 2. 할 일 (Action Items)

| No. | 내용 | 담당자 | 마감 기한 |
|-----|------|--------|-----------|
| A-1 | 모바일 메인 페이지 시안을 디자인 파트에 전달 및 확정 | 팀장 (프로젝트 총괄) | 다음 주 목요일 |
| A-2 | 시스템 연동 API 명세서 작성 완료 | 대리 (시스템 연동 담당) | 다음 달 초 |

## 3. 확인 필요

| No. | 내용 | 비고 |
|-----|------|------|
| C-1 | 보안 규정 관련 인사팀 가이드라인 수령 여부 | 인사팀 가이드라인이 아직 전달되지 않은 상태이므로, 해당 부서에 일정 및 내용 확인이 필요합니다. |
| C-2 | API 명세서 작성 마감 기한의 구체적 일자 | "다음 달 초"로만 언급되어 정확한 마감일이 특정되지 않았으므로, 담당자와 확정 일자 협의가 필요합니다. |
```

### 📄 Few-shot 실제 출력 (PART 2 [1] Raw)

```markdown
# 스마트 스토어 고도화 프로젝트 회의 요약

## 핵심 결정사항

| 번호 | 결정사항 | 비고 |
|:---:|---|---|
| 1 | 모바일 메인 페이지 시안을 디자인 파트에 전달하여 확정 | 다음 주 목요일까지 완료 목표 |
| 2 | 개발 서버 인프라 구축 예산: 기합의 1,200만 원 범위 내 집행 최종 확정 | — |

## Action Items

| 번호 | 할 일 | 담당자 | 마감 기한 | 비고 |
|:---:|---|---|---|---|
| 1 | 모바일 메인 페이지 시안 디자인 파트 전달 및 확정 | 프로젝트 팀장 / 디자인 파트 | 다음 주 목요일 | — |
| 2 | 시스템 연동 API 명세서 작성 | 프로젝트 담당 대리 | 다음 달 초 | 구체적 일자 미확정 — 정확한 마감일 재확인 필요 |

## 확인 필요 사항

| 번호 | 항목 | 상세 내용 | 확인 대상 |
|:---:|---|---|---|
| 1 | 보안 규정 가이드라인 | 상세 보안 규정에 대한 사내 가이드라인이 아직 미수령 상태 | 인사팀 |
| 2 | API 명세서 마감일 특정 | "다음 달 초"로만 언급되어 정확한 날짜(YYYY-MM-DD) 확정 필요 | 프로젝트 담당 대리 |
```

### 📊 항목별 비교 분석표

| 비교 항목 | Zero-shot (v1) 실제 아웃풋 | Few-shot (v1.5) 실제 아웃풋 | 개선 효과 |
| :--- | :--- | :--- | :--- |
| **섹션 제목 구조** | `결정사항(Decision)` / `할 일(Action Items)` / `확인 필요` 3개 섹션 | `핵심 결정사항` / `Action Items` / `확인 필요 사항` 3개 섹션 | 외부 공유에 적합한 한영 혼용 비즈니스 제목으로 자연스럽게 정제됨 |
| **행 인덱싱 방식** | `D-1/D-2`, `A-1/A-2`, `C-1/C-2` 자체 분류 코드 사용 | `1`, `2` 단순 번호 + 중앙 정렬(`:---:`) | Few-shot 예시 양식을 모방하여 가독성 중심의 표준 번호 방식으로 통일됨 |
| **담당자 호칭 정제** | `팀장 (프로젝트 총괄)`, `대리 (시스템 연동 담당)` | `프로젝트 팀장 / 디자인 파트`, `프로젝트 담당 대리` | 외부 발송용에 적합한 역할 중심 호칭으로 한 단계 더 정제됨. 협업 파트(디자인 파트)도 담당자 열에 병기함 |
| **Action Items 표 열 구성** | `No. / 내용 / 담당자 / 마감 기한` (4열) | `번호 / 할 일 / 담당자 / 마감 기한 / 비고` (5열) | `비고` 열이 추가되어 미확정 일정에 대한 주석("구체적 일자 미확정") 처리가 가능해짐 |
| **확인 필요 표 열 구성** | `No. / 내용 / 비고` (3열) | `번호 / 항목 / 상세 내용 / 확인 대상` (4열) | `확인 대상` 열이 추가되어 후속 조치 책임 부서(인사팀, 담당 대리)가 명확히 구분됨 |
| **예산 수치 포함 여부** | 결정사항 D-2에 **1,200만 원** 명시 노출 | 결정사항 2에 **1,200만 원** 명시 노출 | 두 단계 모두 예산 수치 노출 — 보안 필터링은 Turn 3 이후 유저 지시(보안 필터) + CoT v2 적용 시점부터 차단됨 |
| **사족(인사말) 차단** | 완전 차단 — 페르소나 설정 효과로 Zero-shot에서도 `#` 헤더로 바로 시작 | 완전 차단 — Few-shot 예시 1번 패턴 강화로 동일하게 유지됨 | 두 단계 모두 사족 없음. 페르소나 설정이 Zero-shot에서 이미 사족을 억제하는 핵심 역할 수행 |
| **날짜 역산 수행 여부** | 미수행 — "다음 주 목요일", "다음 달 초" 그대로 기재 | 미수행 — 동일하게 상대적 기한 표현 유지 | 두 단계 모두 날짜 역산은 미수행. 캘린더 날짜 추론은 CoT v2의 3단계 지침 추가 이후 시도됨 |

## 6. 최종 선정 결론 (근거 포함)
> **최종 선정 모델**: Claude Opus 4.6

**선정 사유 (최소 3줄 이상 구체적으로 서술):**
1. **압도적인 시각화 및 문서 구조화(가독성)**: 타 모델들이 단순 텍스트 불릿 포인트 요약에 그친 반면, 마크다운 표(Table)와 취소선(~~기존~~ → **변경**)을 자율적이고 적극적으로 활용하여 비즈니스 요약본의 가독성을 최고 수준으로 끌어올렸습니다.

3. **완벽한 10턴 누적 제약 통제 및 사족 차단**: 턴이 쌓여도 예산 제외·가명 처리·이메일 톤·취소선 표기·항목 제거 등 다중 제약 조건이 누락 없이 누적 적용되었으며, 환각 검증 5개 질문 모두에서 임의 수치 창작 없이 올바른 정보 차단 및 후속 조치 안내를 수행하여 **5/5 PASS** 판정을 받았습니다.
