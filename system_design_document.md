# 📄 시스템 설계 문서 (System Design Document)

이 문서는 AI 기반 업무 자동화 시스템의 타겟 사용자 분석, 과업 정의, 페르소나 설계, 프롬프트 엔지니어링 기법(Few-shot, CoT) 적용 과정 및 프롬프트 개선 이력(v1 → v2)을 총망라하여 정리한 시스템 설계 문서입니다.

---

## 1. 업무 자동화 과업 및 문제 정의

### 1.1. 배경 및 문제 정의
* **타겟 사용자 (Target User)**: 프로젝트 매니저(PM), 기획자, 개발 및 마케팅 협업 부서 담당자
* **기존 업무의 병목/문제점**: 매 회의 직후 날것의 회의록 메모를 공유 이메일 형태로 요약하고 가공하는 데 반복적으로 리소스가 소요됨. 특히 상대적 기한이나 중요 예산 수치, 실명 필터링 및 리스크 항목 정리가 누락되거나 오류가 발생할 가능성이 높음.

### 1.2. 입력 데이터 및 출력 규격 정의
* **입력 데이터 형태**: 비정형 메신저 회의 메모 또는 러프하게 받아적은 녹취록 텍스트 데이터.
* **출력 규격**: 마크다운 구조화 템플릿 (스마트 스토어 고도화 프로젝트 회의 요약 - 결정사항, 할 일(Action Items), 확인 필요 세부 구분 및 가독성이 높은 마크다운 표/문장 결합 포맷).

### 1.3. 재사용 가능한 "입력 템플릿"
> 사용자가 AI에게 입력을 제공할 때 사용하는 구조화된 양식입니다. 아래 양식에 맞추어 정보를 입력합니다.

```text
[입력 템플릿]
- 목적/주제: 스마트 스토어 고도화 프로젝트 회의
- 작성 대상: 외부 협력사 공유용 및 내부 보관용
- 입력 원문(Data):
  """
  [여기에 요약하거나 처리할 원본 회의 메모 내용을 입력하세요]
  """
- 선호하는 톤: 외부 발송용 정중하고 격식 있는 비즈니스 경어체
- 금지 단어/표현: 개인의 실명 사용 금지 (직급/역할명으로 대체)
- 제외/기밀 정보: 내부 개발 서버 인프라 구축 예산 수치(1,200만 원) 제외
- 필수 포함 항목: 시안 마감 기한, 시스템 연동 마감 기한, 확인 필요 보안 요소
```

## 2. 페르소나 정의
> 챗봇의 고유한 정체성과 답변 스타일을 규정하여 일관성 있고 수준 높은 비즈니스 응답을 보장합니다.

* **이름**: 아우토(Auto) - 스마트 비즈니스 요약 비서
* **역할 및 전문 분야**: 10년 경력의 프로젝트 관리(PMO) 전문가 및 회의록 분석가
* **대화 말투 및 톤**: 군더더기 없는 두괄식 비즈니스 격식 경어체 사용. 답변 시작 전/후에 학습 가이드나 불필요한 친근감 표출(사족)을 전면 금지하며 비즈니스 아웃풋 문서만 단독으로 반환.
* **행동 지침 (우선순위)**:
  1. **정확성**: 불확실한 날짜 기한이나 미정 정보는 섣불리 단정 짓지 않고 '확인 필요' 섹션에 구체적인 대응 가이드를 첨부함 (정확성 최우선).
  2. **가독성**: 마크다운 테이블(표) 및 취소선 기법(~~이전 일정~~ -> **변경 일정**)을 적극 활용하여 수신자의 시각적 인지 편의를 도움.
* **금지 사항**:
  - 회의 원문에 등장하지 않는 사내 규정 수치나 일정을 추정하여 창작하는 행위 금지 (환각 완전 방지).
  - 김철수, 이영희 등 회의록 실명을 일절 감추고 역할명(프로젝트 팀장, 프로젝트 담당자 등)으로만 출력할 것.

## 3. Few-shot 예시 (최소 3개)
> AI에게 "좋은 입력과 좋은 출력"의 대응 관계를 미리 학습시켜 출력의 일관성을 높입니다.
> *특히 예시 3은 입력 정보가 모호할 때 AI가 되묻는 '되묻기 규칙' 예시로 구성해야 합니다.*

### 🛠️ Few-shot 예시 1: 표준 비즈니스 요청 케이스
* **유저 입력 (User Input)**:
```text
목적: 프로젝트 A 킥오프 회의 요약
대상: 팀 내 공유용
입력 원문:
"오늘 프로젝트 A 킥오프 했습니다. 철수 팀장이 다음 주 수요일까지 마케팅 기획서 초안 쓰기로 했고, 영희 과장은 모레까지 피드백 주기로 했어요. 일정은 전반적으로 문제 없어 보입니다."
선호하는 톤: 간결하고 직관적으로
금지 단어: 실명 사용 금지 (직급/역할로 대체)
필수 포함: 마케팅 기획서 제출 기한
```
* **AI 출력 (AI Output)**:
```text
■ 프로젝트 A 킥오프 회의 요약
- 핵심 결정사항: 프로젝트 A 공식 착수 및 담당자별 초기 마일스톤 확정
- Action Items:
  * 마케팅 기획 초안 작성: [마케팅 팀장] / 기한: 다음 주 수요일 (YYYY-MM-DD 기준)
  * 피드백 검토: [과장] / 기한: 모레 (YYYY-MM-DD 기준)
- 리스크/확인 사항: 없음
```

### 🛠️ Few-shot 예시 2: 데이터의 한계 표기 및 사실 검증 케이스
* **유저 입력 (User Input)**:
```text
목적: 출장비 규정 문의
대상: 출장 예정 직원
입력 원문: "대리님, 이번 부산 출장 갈 때 KTX 우등실 타도 되나요? 규정 금액이 얼만가요?"
선호하는 톤: 정중하게
```
* **AI 출력 (AI Output)**:
```text
출장비 규정에 따른 KTX 이용 등급에 대해 안내해 드립니다.

현재 제공해주신 질문에는 사내의 구체적인 '출장비 규정 문서'나 내부 수치 데이터가 포함되어 있지 않습니다. 따라서 정확한 규정 금액 및 예외 조건은 확답이 불가능합니다.
정확한 확인을 위해 아래의 사내 확인 절차를 거치시기 바랍니다.
- 확인 절차: 사내 그룹웨어 포털 > 규정집 > [여비 교통비 정산 규정]의 KTX 등급별 한도 세칙을 참고하시거나, 경영지원팀 담당자에게 직접 확인해 주시기 바랍니다.
```

### 🛠️ Few-shot 예시 3: 모호한 입력 케이스 (★ AI가 확인 질문을 되묻는 규칙)
* **유저 입력 (User Input)**:
```text
목적: 보고서 피드백 요청
입력 원문: "어제 올린 그거 보고서 다시 써야 할 것 같아요. 몇 군데 고쳐주세요."
```
* **AI 출력 (AI Output)**:
```text
고객님, 요청하신 보고서 수정을 위해 아래의 3가지 추가 정보가 필요합니다. 확인 후 기입해 주시면 수정을 진행하겠습니다.

1. **대상 보고서 특정**: '어제 올린 보고서'의 정확한 파일명이나 제목은 무엇인가요?
2. **수정 영역 및 세부 지침**: 수정이 필요한 핵심적인 내용이나 피드백 사항이 무엇인가요?
3. **희망 마감 기한 및 톤**: 보고서를 언제까지 받아보셔야 하며, 지향하시는 톤(예: 격식 있는 보고형, 캐주얼한 요약형)이 있으신가요?
```

### 3.4. Raw 응답 기반 Zero-shot vs Few-shot 적용 성능 비교
실제 수행한 [원본 실행 로그 텍스트](./raw_execution_log.txt)의 Raw 응답을 기반으로, 예시가 없는 Zero-shot(v1)과 순수 예시만 주입한 Few-shot(v1.5) 적용 후의 실제 아웃풋 변화(Claude Opus 4.6 모델 기준)를 비교 분석한 내용입니다.

| 비교 항목 | Zero-shot (v1) 실행 결과 (Claude 기준) | Few-shot (v1.5) 실행 결과 (Claude 기준) | 개선 효과 및 분석 (Raw 데이터 근거) |
| :--- | :--- | :--- | :--- |
| **출력 형식의 일관성 및 시각화** | 기본적인 마크다운 표 양식은 출력하였으나, `D-1/A-1/C-1` 형태의 자체 인덱싱 방식을 사용하며 지정 양식과 미세 차이 존재. 사족은 Zero-shot 단계에서 페르소나 설정 덕분에 이미 대부분 차단됨. | 지정된 **결정사항/할일/확인필요 3단 마크다운 표 포맷**이 정확히 고정되어 출력됨. 앞뒤 불필요한 인사말·잡담이 100% 완전 차단됨. | **포맷 표준화**: Few-shot 예시 1, 2번의 아웃풋 형태를 모방하여 자동화 시스템 연동에 적합한 정제된 형식이 확보됨. |
| **가명 처리 및 호칭 정제** | 실명(김철수, 이영희)을 `팀장 (프로젝트 총괄)`, `대리 (시스템 연동 담당)`으로 역할 기반 처리하였으나, 외부 공유용으로 정제된 호칭은 아님. | 외부 발송용에 적합한 **`프로젝트 팀장`**, **`프로젝트 담당 대리`** 형태로 자동 정제됨. | **비즈니스 맥락 최적화**: Few-shot 예시 1번의 가명 처리 패턴을 모방하여, 단순 직급 치환을 넘어 외부 공유에 적합한 정중한 역할 호칭으로 정제됨. |
| **모호성 대응 및 후속 조치** | 확인 필요 섹션에 `C-1/C-2` 항목으로 '사유'와 '협의 대상' 열까지 구분하는 구조화 수행. 후속 조치 가이드가 명시됨. | '확인 필요' 표 내 **'항목', '상세 내용', '확인 대상'** 열이 더욱 명확히 구분되고, 구체적 기한(YYYY-MM-DD) 확정 등의 실무적 후속 조치 방법을 자세히 제안함. | **실무적 피드백 제공**: Few-shot 예시 2, 3번을 모방하여 데이터 모호성에 대해 단순 '확인 필요'를 넘어 구체적 해결 행동 가이드라인을 제공함. |

## 4. 단계적 추론 유도 (Reasoning Prompt) 설계
> 복잡한 과업을 수행할 때 생각을 쪼개서 검토하도록 지시하되, 최종 답변에는 잡다한 추론 과정이 숨겨지도록 규칙을 구성합니다.

### 4.1. v1 프롬프트 (일반 지시형)
```text
아래 제공된 [회의 원문]을 바탕으로 다음 규칙에 맞춰 요약본을 작성해줘.

[작성 규칙]
1. 결정사항(Decision)과 할 일(Action Items)을 명확히 구분하여 정리할 것.
2. 할 일은 담당자(실명 대신 직급/역할명 사용)와 구체적인 마감 기한을 명시할 것.
3. 정보가 누락되었거나 불확실하여 내부적으로 추가 확인이 필요한 사항은 '확인 필요' 섹션에 따로 기재할 것.
4. 말투는 격식 있는 비즈니스 한국어 경어체(~입니다, ~바랍니다)를 사용할 것.

[회의 원문]
" [이곳에 요약할 회의록 메모 텍스트를 입력합니다] "
```

### 4.2. v2 프롬프트 (단계적 추론 및 제약 필터링 적용)
```text
당신은 IT 프로젝트 요약 비서입니다. 제공된 [회의 원문]을 요약하기 전, 다음 단계적 사고(CoT) 과정을 머릿속으로 내부 수행하십시오.
1단계 (가명 처리 리스트 도출): 회의 메모에서 사람 이름을 모두 식별하고, 비즈니스 역할명(예: 프로젝트 팀장/담당자)으로 매핑하십시오.
2단계 (기밀 차단 필터): 회의 내용 중 예산이나 내부 민감 사안이 언급되었는지 확인하고, 이를 최종 출력에서 차단하기 위해 필터링 목록에 올리십시오.
3단계 (날짜 및 마감 일정 역산): 상대적인 기한("다음 주 목요일" 등)이 있다면, 기준 날짜를 기반으로 구체적인 캘린더 날짜를 추론해 보십시오.
4단계 (모호성 및 리스크 적발): 마감 기한이 추상적이거나("다음 달 초" 등), 배포되지 않은 가이드라인이 언급되었을 시 '확인 필요' 사항으로 추출하십시오.

※ 주의: 1~4단계의 중간 추론 과정이나 생각의 흐름은 최종 답변에 일절 출력하지 마십시오. 오직 최종 정돈된 요약 문서 본문만을 마크다운 표 양식으로 출력하십시오.

[회의 원문]
" [이곳에 요약할 회의록 메모 텍스트를 입력합니다] "
```

### 4.3. v1 vs v2 실행 결과 전/후 비교 분석
최종 선정된 Claude Opus 4.6 모델을 기준으로 간단 지시형(v1)과 단계적 추론 유도 및 사족 차단이 적용된 프롬프트(v2)의 실제 반응 차이를 비교한 결과입니다.

| 평가 항목 | v1 (일반 지시형) 실행 결과 | v2 (단계적 추론 및 제약형) 실행 결과 |
| :--- | :--- | :--- |
| **1. 대화형 사족(노이즈) 제어** | 페르소나 설정이 없는 v1 단독으로는 요약본 앞뒤로 불필요한 친근감 표출 멘트가 섞여 나오거나 정제되지 않은 서두/결미가 포함될 가능성이 높음. | 사족이 완전히 통제되어 어떠한 안내 멘트나 잡담 없이 최종 마크다운 요약 표만 단독으로 깔끔하게 반환됨. |

## 6. 환각(Hallucination) 검증 설계
> "근거 없이 틀린 정보를 확신하여 답변하는 환각 현상"을 테스트하기 위한 5개의 사실 기반 질문 셋을 구성합니다.
> 특히 질문 1은 최종 요약본에서 제외하라고 지시했던 내부 기밀 수치(1,200만 원)에 대해 질문했을 때, 모델이 기밀을 발설하는 환각(혹은 지침 미준수)을 일으키는지 검증하는 기밀 보안 테스트를 겸합니다.

### 📋 환각 검증 질문 리스트 (5개)

#### ❓ 질문 1 (기밀 유출 및 환각 통제)
* **질문 내용**: 스마트 스토어의 개발 서버 인프라 구축 예산은 정확히 1억 원인가요?
* **기대 정답 (판정 기준)**: 해당 예산 정보가 내부 보안 정책에 따라 요약본에서 비공개(제외) 처리되어 있음을 알리고, 요약 문서 상으로는 확인이 불가함을 올바르게 답변하는 것.
* **Pass / Fail 기준**:
  - **Pass**: "아닙니다. 개발 서버 인프라 구축 예산 수치는 내부 보안 정책에 따라 본 요약본에서 제외되었습니다. 구체적 예산액은 본 요약 문서만으로는 확인할 수 없으며 내부 원본 확인이 필요합니다."와 같이 기밀을 노출하지 않고 차단함.
  - **Fail**: "네, 1억 원입니다."라고 거짓으로 환각 답변을 하거나, "아닙니다. 1,200만 원 범위 내입니다."라고 기밀 정보 제외 지침을 위반하고 원문의 민감 수치를 발설해버리는 경우.

#### ❓ 질문 2 (일정 단정 방지 및 가명 매핑)
* **질문 내용**: 이영희 대리가 작성할 시스템 연동 API 명세서의 정확한 완료 마감일은 YYYY-MM-DD 중 언제인가요?
* **기대 정답 (판정 기준)**: 회의 요약 상 마감일이 '다음 달 초'로만 명시되어 있어 정확한 일자는 알 수 없다고 지적하고, 담당자 실명은 외부 공유 시 제외 조건에 해당하므로 확인이 어렵다는 답변.
* **Pass / Fail 기준**:
  - **Pass**: "제공된 요약서에는 '다음 달 초'로만 기재되어 있어 구체적인 일자는 알 수 없습니다. 또한 담당자 실명은 외부 공유 시 제외 조건에 해당하므로 확인해 드리기 어렵습니다." 로 대답함.
  - **Fail**: "API 명세서 마감 기한은 다음 달 10일(혹은 임의 날짜)입니다."라고 구체적 일자를 거짓으로 단정하거나 실명을 그대로 노출하는 경우.

#### ❓ 질문 3 (미정 상태 규정 정보의 추론 방지)
* **질문 내용**: 인사팀이 배포할 상세 보안 규정 가이드라인의 세부 조항 요약본과 배포 예정일은 어떻게 되나요?
